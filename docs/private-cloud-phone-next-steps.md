# 私有化云手机（x86 / 大内存）— 后续方案与执行清单

本文承接对话中已对齐的前提：

| 项 | 取值 |
|----|------|
| 部署形态 | **私有化**，**x86**，单机 **128G～256G** 内存，追求 **一机多开**、**高翻台** |
| 流量目标 | 每小时 UV **最高约 20 万**；**平均会话约 3 分钟** |
| 并发量级（规划用） | 均匀到达粗算平均并发 ≈ `UV/h × 0.05`；**晚高峰建议按平均的 2～3 倍** 预留 → **约 1 万～3 万** 路「同时在云」会话（非 20 万路并行） |
| 产品方向 | **必须原生掌厅 APK**（非 H5 替代）；浏览器经 **WebRTC+SFU** 看云机画面并可操作；**触控 C2** 不单独作为扩缩容杠杆 |
| **本机仓库根（Mac）** | **`/Users/mac/程序/cloudPhone/`** —— 根目录 `docker-compose.yml` 起 Redroid 时：`cd /Users/mac/程序/cloudPhone && docker compose ps`（应见 **`cloudphone-redroid`** 为 **Up**；路径随你本机克隆位置调整） |

**与仓库关系**：`experiments/webrtc-sfu-pilot`（Layer B / C1）用于 **媒体与信令验证**；根目录 **Redroid** 用于 **安卓实例**；**商用密度与调度** 需在控制面与会话层单独建设。

**技术栈冻结（团队决议）**：**坚持现有路线，不更换主视频栈。** 浏览器出画 **统一 WebRTC + mediasoup SFU（`consume`）**；云机画面进入 SFU **统一经已跑通的 C1 一脉（PlainTransport ingest / 等价 ingest）**；**不**把 **WSS + 独立投屏解码（文档中的 3a 主链路）** 引入为与 SFU 并行的主出画方案。触控 **C2**（如 `adb shell input`）与 **信令/HTTPS 入口** 可在该栈上迭代，**不改变**上述媒体主栈。

---

## 一、总原则（避免走偏）

### 1.1 实施总线（三条，团队对齐以此为准）

1. **第一步：先在 ECS 上彻底跑通** —— 用户通过 **链接（HTTPS/WSS）** 访问 **云化掌厅**（可看、可完成既定操作路径；触控若产品必需则在本步一并打通）。**本步完成前，不把「本地单机压测 / 集群压测」当作阻塞项。**
2. **第二步：再做单机压测** —— 在 **本地（机房/实验室）大内存 x86 目标机** 上测 **一机多开** 与 **N_stable**（与 ECS 规格不同，**不以 ECS 替代本步**）。
3. **第三步：最后做集群化压测** —— 在 **多机、池化、与目标调度策略** 下做 **压测与容量/成本验证**（承接 UV、翻台、背压等）。

### 1.2 配套原则（在总线各步内部仍要遵守）

- **媒体与调度解耦（在栈内）**：会话分配、鉴权、**C2 触控 API** 与 **WebRTC 媒体面** 接口清晰；**不在此名义下引入第二套主出画协议**（与上文 **技术栈冻结** 一致）。
- **掌厅等加固 App**：以 **实测** 为准；与 SFU 是否通无关。
- **第一步（ECS）内** 允许 **最小控制面**（例如单台 ECS 上少量实例、简单分配/回收），**完整**会话池、多机路由、大规模排队策略可放到 **第三步** 与集群压测一起收紧。
- **指标仍要记**：第一步跑通过程中就要记 **首帧、失败率、单会话资源**，为第二、三步提供输入，但 **不颠倒上述三条顺序**。

## 二、分阶段方案

**与 §1.1 的对应关系**：**里程碑第一步** ≈ 下文的 **阶段 0（与体验相关的子集）+ ECS 上的 Redroid/掌厅 + 阶段 3 + 阶段 4（按需）+ 最小实例分配**；**里程碑第二步** ≈ **阶段 1b**；**里程碑第三步** ≈ **阶段 2 + 阶段 5** 及专项 **集群压测**。

### 阶段 0 — 需求与指标定稿（约 1 周内可完成）

**要做什么**

- [ ] 统一口径：**UV** 定义（设备 / 账号 / 业务会话）；是否计 **重复进入**。
- [ ] 写清 **SLO**：例如首帧 P95、单会话最大时长、允许排队时长、会话失败重试策略。
- [ ] 确定 **峰值系数**（晚高峰相对均匀到达的倍数，建议先取 **2～2.5**，预留到 **3**）。
- [ ] 确定 **单路视频目标**（分辨率、码率上限），用于带宽与编码规划。

**产出**：一页 **《容量与体验 SLO》**（内部即可）。

---

### 阶段 1 — 单机密度压测（目标 App；**属 §1.1 第二步**）

**1a — ECS（纳入「第一步：链接访问云化掌厅」，不作为 N_stable 依据）**

- [ ] 在 **ECS** 上跑通 **Redroid + 掌厅 + 链接访问** 的端到端；开发期仍可用 **`scripts/layer-c0-redroid-on-ecs.sh`** + SSH 转发 **`adb` / `scrcpy`** 排障（`docs/layer-c-roadmap.md` §5）。
- [ ] 与 **`webrtc-sfu-pilot` / C1** 等按需联调（媒体链验证）。

**1b — 本地大内存 x86（**里程碑第二步**：权威 N_stable）**

- [ ] 在 **私有化目标硬件**：**本地（实验室/机房）128G～256G x86 真机** 上部署与线上一致的 **Redroid compose**（可与 ECS 用同一套 compose，但**以本机内核/CPU/内存为准**）。
- [ ] 安装 **与线上一致的官方 APK**（如掌厅，见 `docs/redroid-notes.md`）。
- [ ] **单实例** 稳定路径：`adb` + `scrcpy`（或你们选定的 Web 预览）；**不依赖** ECS 公网路径做密度结论。
- [ ] **加压**：同机递增实例数（10 → 20 → …），每档跑 **典型 3 分钟任务**，记录：
  - 容器 **CPU / 内存**（`docker stats` / cgroup）
  - **卡顿、崩溃率**、冷启时间
  - 必要时 **`adb shell dumpsys meminfo <包名>`**（见 `docs/linux-cloud-lab.md` 相关小节思路）

**产出**：**《单机最大稳定实例数 N_stable》** —— **以 1b 本地真机数据为准**；ECS 上若做对比压测，仅作参考，**不替代 1b**。

**验收**：能给出「在 SLA 下，**单机 N**（来自 **1b**）；峰值并发 **M** 时，**至少 ceil(M/N) 台** 私有化机器」的数量级。

---

### 阶段 2 — 会话与资源调度（控制面 MVP；**属 §1.1 第三步**为主，第一步仅需最小可用子集）

**要做什么**

- [ ] **实例池**：创建 / 预热 / 健康检查 / **回收**（会话结束、超时、异常）。
- [ ] **分配策略**：先进先出、最短队列、亲和性（同用户复用实例可选）。
- [ ] **翻台**：**空闲超时**、**最大会话时长**、与 **3 分钟均值** 对齐的产品规则。
- [ ] **背压**：池满时 **排队 + 提示** 或 **降级**（非无限接入）。
- [ ] **观测**：分配耗时、池深度、失败原因、每机实例数。

**产出**：可对接任意「画面出口」的 **调度 API**（HTTP/gRPC 均可）+ 基础 **Dashboard 或日志查询**。

**验收**：无真实用户时，用脚本模拟 **高到达率** 请求，池行为符合策略、无泄漏。

---

### 阶段 3 — 浏览器访问链路（MVP；**属 §1.1 第一步**核心）

**要做什么**

- [ ] **选型（二选一为主链路，勿双栈并行）**  
  - **3a**：Web 投屏类方案，浏览器 **WSS + 解码**，服务端连 **ADB/设备**（与 **WebRTC+SFU 出画** 是 **另一条视频栈**，主链路已走 SFU 时 **不应再作为主出画方案**）。  
  - **3b（与现有 SFU 统一）**：屏幕 → **H264 RTP → mediasoup PlainTransport** → 浏览器 **WebRTC consume**（在 C1 已通基础上换源），见 `docs/layer-c-roadmap.md` C1。  
  - **决策**：若团队 **已通过 WebRTC+SFU 完成浏览器出画**，则 **主技术路线即 3b 一脉**；后续只做 **ingest 源质量、产品页、HTTPS、鉴权、C2**，**不必**为「统一路线」再引入 **3a**（除非明确要 **降级备用客户端** 或 **对照调试**，且接受双栈成本）。
- [ ] **TLS**：私有化域名或内网证书；**WSS/HTTPS** 与 `docs/aliyun-ecs-pilot.md` §4.2 思路一致（可 Caddy/Nginx）。
- [ ] **鉴权**：Token / 会话绑定，**禁止**仅靠保密 URL。

**产出**：**端到端**：浏览器打开链接 → 获得一台池内实例 → **可稳定看画面**（触控可阶段 4）。

**验收**：小规模用户（或压测机器人）**重复翻台**，无实例泄漏、无明显资源爬升。

---

### 阶段 4 — 触控与体验（C2；**属 §1.1 第一步**若产品要求「可操作掌厅」）

**要做什么**

- [ ] **坐标映射**：分辨率、DPI、横竖屏与浏览器坐标一致。
- [ ] **回注通道**：`adb shell input` 或应用内协议（与 `docs/layer-c-roadmap.md` C2 一致）。
- [ ] **限频与防滥用**：防止脚本打爆 `input`。
- [ ] **（生产前架构项）触控 API 独立服务化**：测试期可先在 **`webrtc-sfu-pilot` / `server.cjs` 同进程** 提供回注接口以提速验证；**上线生产前**将 **`adb shell input` 等控制面** 拆为 **独立服务**（单独进程/容器、独立鉴权与限流、与 SFU 解耦），理由与阶段划分见 **`docs/layer-c-roadmap.md`** — C2 小节「前期与后期——架构备忘」。

**产出**：与阶段 3 同一套会话内的 **可靠点击/滑动**。

**验收**：典型 3 分钟任务路径 **人工可完成**，误触率在可接受范围。

---

### 阶段 5 — 横向扩展与成本（**属 §1.1 第三步**；含集群化压测）

**要做什么**

- [ ] **多机分池**：按机架 / 机房划分池，调度层 **路由到池**。
- [ ] **弹性**：低峰缩容、高峰扩容（或固定池 + 排队策略明确）。
- [ ] **带宽**：按「峰值并发 × 单路码率」核算 **出口与内网**；避免单机网卡先爆。
- [ ] **编码**：评估 **硬件编码** 节点与 **云机节点** 是否分离（成本与延迟权衡）。
- [ ] **集群化压测**：在多机池、目标调度与背压策略就绪后，按 UV/并发目标做 **专项压测** 与瓶颈报告（与单机 **1b** 区分）。

**产出**：从 **「ceil(峰值并发 / N_stable)」** 到 **「多池 + 带宽 + 冗余」** 的 **拓扑与采购清单（量级）** + **集群压测结论**。

| 顺序 | 里程碑 | 做什么（摘要） |
|------|--------|----------------|
| **①** | **ECS：链接 → 云化掌厅跑通** | 阶段 **3**（HTTPS/WSS、鉴权、Web 投屏或选定媒体路径）、ECS 上 **Redroid + 掌厅**、阶段 **4**（若必须可操作）、阶段 **2** 的 **最小子集**（能分配/释放当前 ECS 上的实例即可）；阶段 **0** 中与体验/SLO 相关的条目并行补齐 |

## 六、附录：在 **C0 + C1 已跑通** 的前提下 —— 到「**ECS 链路透掌厅**」还要做什么

**本文评估基准（你已具备）**

- **C0**：ECS 上 **Redroid**、**ADB（含 SSH 转发）**、**scrcpy 可见云机画面** 已验证（`docs/layer-c-roadmap.md` §5）。  
- **C1**：**`PlainTransport` ingest → `produce` → 浏览器 `consume`** 已在 ECS 相关环境跑通（含 **安全组 UDP、`MEDIASOUP_ANNOUNCED_IP`、ingest 与 Docker 镜像同步** 等，见 `experiments/webrtc-sfu-pilot/docs/layer-c1-lessons-learned.md`）。

**在此基础上的「剩余差距」**（不等于从零做安卓或从零做 SFU）

- **公网产品链**：**`https://…` 链接**、**WSS（若走 3a）** 或 **继续用现有 WebRTC 观看链（若走 3b）**、**鉴权与实例绑定**，与「本机 scrcpy 排障」不是同一套交付。  
- **掌厅**：若 C0 阶段仅在桌面验证，仍需 **官方 APK + 业务路径在云机内稳定**（`docs/redroid-notes.md`）。若已装且路径已通，**包 A** 以验收补文档为主。  
- **画面源选择**：**3a** 与 **3b** 二选一或先后（对比见 **§6.5**，已按 **已有 C0/C1** 修正表述）。  
- **触控（可选）**：阶段 4，与媒体方案并行设计接口即可。

下面按 **推荐实施顺序** 列计划（可并行标出）。

### 6.1 差距一览（你「还要补」的块）

| 块 | 说明 |
|----|------|
| **云机** | **C0 已覆盖**：Redroid + ADB + 画面可见。**剩余**：若走 **3a**，需 **与 Redroid 同机/同网** 的 **Web 投屏进程** 能访问 `127.0.0.1:5555`（或你们实际 ADB 口）；若走 **3b**，需 **采集侧** 能访问同一 ADB/设备 |
| **掌厅** | **在 C0 已通设备上**：若尚未装 → **官方 APK** + 冷启与 **3 分钟级业务路径**（`docs/redroid-notes.md`）；若已通 → 本块主要为 **链路透浏览器后的回归** |
| **链路透浏览器** | 在 **公网/域名** 下 **HTTPS + WSS**（`docs/aliyun-ecs-pilot.md` §4.2）；**不是**仅 SSH + scrcpy 桌面 |
| **画面源** | **二选一为主链路**：**3a** WSS+投屏解码；**3b** 真屏 → **H264 RTP → 复用 C1 ingest** → **WebRTC consume**。若 **已 WebRTC+SFU 出画**，则 **主链路即 3b**，勿再叠 **3a** 作主出画 |
| **链路与实例** | **最小控制面**：打开链接 / 带 Token → 绑定 **当前这台** Redroid（首版可 **固定 1 台** 实例，不做多池） |
| **可操作（可选）** | 若验收要求「能点掌厅」：阶段 **4**（坐标 → `adb shell input` 或等价） |
| **观测** | 首帧耗时、断线重连、单会话 CPU/内存，便于后续单机压测与集群压测 |

### 6.2 推荐排期（工作包顺序）

**包 A — 云机 + 掌厅（C0 已跑通时的最小补漏）**

- [ ] 确认 **C0 路径仍可作为排障后备**（`scripts/layer-c0-redroid-on-ecs.sh` + SSH + `scrcpy`）。  
- [ ] **掌厅**：若未装 → `adb install` 官方包 + **scrcpy 下**冷启与关键页（`docs/layer-c-roadmap.md` §5）；若已装 → 写清 **版本 + 复现步骤** 供链路透浏览器后对照。  
- [ ] 记录：**Redroid 镜像标签、Android API、掌厅版本**。

**包 B — 媒体主方案（与 C1 的关系写清楚）**

- [ ] **评审定案（按当前事实分支）**  
  - **已 WebRTC+SFU 出画**：**主链路 = 与 C1 同族的 ingest + consume**（文档中的 **3b**），**不要**再把 **WSS+投屏解码（3a）** 当作主出画——否则 **双视频栈**，运维与端口/带宽成本重复。  
  - **尚未有浏览器出画、只有 C1 彩条**：可 **先 3a** 打通「链接见掌厅」，再视需要迁 **3b**；或 **直接 3b**（见 §6.5）。  
- [ ] **若选 3a（且与 SFU 并存仅作例外）**：Web 投屏服务 + **WSS** + **ADB→设备**；单独评估 **与 mediasoup UDP/端口** 的隔离。  
- [ ] **若选 3b / 已走 SFU 出画**：**不改观看主路径**，聚焦 **ingest 侧车**（采集 → 编码 → RTP）与 **掌厅/加固**（`docs/layer-c-roadmap.md` §3）。

**包 C — 链接与 TLS（对最终用户形态）**

- [ ] 域名 **`A`** → EIP；**Caddy**（或 Nginx）**HTTPS**，证书自动或手动。  
- [ ] 浏览器用 **`https://域名/...`** 打开 **投屏页**（或你们封装页），**非** `http://裸 IP` 凑合（摄像头/安全上下文仍参考 `docs/webrtc-sfu-pilot.md` §3.2；投屏页以实际方案为准）。  
- [ ] **鉴权**：首版至少 **Token / 路径参数 + 服务端校验**，避免裸链。

**包 D — 最小控制面（不做大池也可验收①）**

- [ ] 首版可 **单 Redroid 实例**：链接只做 **鉴权 + 路由到该实例**；会话结束 **回收/超时** 逻辑可简化为 **断开连接 + 可选重启容器**。  
- [ ] 文档写明：**当前版本并发 = 1**；多会话再迭代。

**包 E — 触控（按产品验收决定是否进①）**

- [ ] 若必须：浏览器事件 → **HTTP/WebSocket API** → **`adb shell input`**（阶段 4）；与 **包 B 画面延迟** 一起测。

### 6.3 「ECS 链路透掌厅」建议验收标准（书面打勾）

- [ ] **外网浏览器**（非 SSH 隧道本机独占）打开 **`https://…`** 后 **≤ X 秒** 看到 **云机内掌厅界面**（X 由阶段 0 SLO 填）。  
- [ ] 完成 **一条既定业务路径**（与产品对齐，例如：冷启 → 首页 → 某二级页）。  
- [ ] 若要求可操作：**该路径内点击/滑动** 可用。  
- [ ] **断线重连** 或 **刷新页面** 行为可接受（首版可允许「需重新取链」）。  
- [ ] **同 ECS** 上 **C1 能力保留或明确下线方式**（避免两套媒体争端口/争 UDP；若同机部署，写清端口与 `network_mode`）。

### 6.4 与仓库任务的对应（便于拆 issue）

| 工作包 | 主要落点 |
|--------|----------|
| A | 根目录 **`docker-compose.yml`**、`docs/redroid-notes.md`、`scripts/layer-c0-redroid-on-ecs.sh` |
| B-3a | 新服务或 `experiments/ws-scrcpy` 等 + ECS compose/进程管理 |
| B-3b | `experiments/webrtc-sfu-pilot/server.cjs` ingest + 采集侧车 / 管道 |
| C | `experiments/webrtc-sfu-pilot` 的 **Caddy** 编排思路 → 投屏域名反代；`docs/aliyun-ecs-pilot.md` §4.2 |
| D | 新建小服务或试点页扩展（会话 ID、Token） |
| E | 独立 **input API** + 与前端坐标约定 |

### 6.5 **3a** 与 **3b** 工作量对比 & **3a → 3b** 迁移（**前提：C0 + C1 已跑通**）

以下量级 **相对「从零做安卓 + 从零做 SFU」已缩短**：C0 吃掉 **设备可见性与 ADB** 风险，C1 吃掉 **ingest + produce + consume + ECS 媒体面** 风险。仍随 **Web 投屏方案选型 / 采集编码实现** 波动。

| 维度 | **3a（Web 投屏）** | **3b（真屏 → RTP → 已有 C1 ingest）** |
|------|---------------------|----------------------------------------|
| **核心工程** | **增量几乎全在新栈**：**WSS + 设备侧拉流**（ADB/scrcpy 协议侧），浏览器 **解码/渲染**；与 C1 **可并行共存** | **增量集中在「侧车」**：**持续采集 + 稳定 H264 + RTP 注入** 以 **替换 FFmpeg 彩条**；**mediasoup ingest / Router / 浏览器 consume** 你们已通，**不必从零证 SFU** |
| **与 C0/C1** | **C0**：设备与排障已熟；**C1**：可作技术备份或与 3a **同机注意端口/UDP** | **C0**：采集一般仍经 **ADB/同机**；**C1**：ingest 参数、RTCP mux、镜像更新等 **直接复用已踩坑结论**（见 layer-c1-lessons-learned） |
| **浏览器** | **新产品页**：多为 **WebCodecs/MSE/Canvas** 等（依 3a 方案） | **可基于现有试点页演进**：`RecvTransport` + `consume` **路径已通**；产品化多为 **抽组件、换 UI、接会话** |
| **网络** | 以 **TCP/WSS** 为主（另放行业务端口） | **观看端 UDP** 与 **C1 已配置项一致**；ingest 仍在 **服务端本机/侧车** |
| **工作量量级（粗，在 C0+C1 后）** | 仍常见 **人周级～约 1 人月** 单路 MVP（**不含**你们已完成的 C0/C1） | **相对无 C1 团队显著缩短**：**人周级～数人周** 到「能替换彩条、看见真屏」常可争取；**达到掌厅稳定 + 加固兼容** 仍可能 **拖到人月级**（不确定块在 **App/录屏** 而非 mediasoup） |

**3a → 3b 后期迁移「方便吗」？**（与是否已完成 C1 **无关**，结论同前）

- **易保留**：会话、Token、HTTPS 入口、ECS/compose、**adb 触控 API**、观测规范（建议 **媒体入口描述** 接口化）。  
- **仍属换代**：**浏览器出画**（3a 解码栈 ↔ 3b WebRTC consume 产品页）；**服务端设备→比特**（WSS 桥 ↔ 编码+RTP ingest）。**不是改配置即切**。  
- **你们多一项利好**：迁 3b 时 **不必再证一遍「consume 能否通」**，主要证 **新源 + 运维**。

**在 C0+C1 基础上的结论**：**3b 相对「无 C1」已明显减负**（ingest/观看链已通）。**若 additionally 已通过 WebRTC+SFU 完成真机出画**：**坚持单栈**，后续以 **ingest 质量 + 链路与掌厅 + C2** 为主，**不必**再上 **3a** 作为主路线（与「是否先 3a」的旧建议不矛盾——旧建议针对 **当时仅有 C1 彩条、尚无 SFU 真屏** 的情形）。**3a→3b** 的迁移叙述，仅适用于 **曾选 3a 作主出画** 的团队。

完成 **§6.3** 全部打勾，即视为 **§1.1 第一步** 可书面结项；再进入文档 **§1.1 第二步（1b）**。

## 七、固定技术栈下的执行计划（**仅** WebRTC + mediasoup SFU **`consume`** + **C1 一脉 `PlainTransport` ingest**）

与文首 **「技术栈冻结」** 一致：**不**新增 WSS 投屏作为主出画。以下顺序可按迭代排期；标 **可并行** 的与主线同时推进。

### 7.1 媒体面（ingest → Router → consume）

| 顺序 | 工作项 | 要做什么 | 产出 / 验收 |
|------|--------|----------|-------------|
| **W1** | **ingest 源稳定** | 侧车或管道：**云机真屏 → 持续 H264 编码 → RTP**，参数与当前 **`server.cjs` ingest**、**`MEDIASOUP_INGEST_RTCP_MUX`、PT/SSRC** 等对齐；替换 FFmpeg 彩条后 **长时间无断流** | **仓库已提供 PoC**：`scripts/ffmpeg-ingest-h264-adb-screenrecord.sh`；开工前 **`npm run c1:check:adb`**；推流 **`npm run c1:ingest:adb -- --local`**（`server.cjs` 启动日志亦提示此项）。验收：`rtpBytesReceived` 稳定涨；浏览器 consume **见掌厅画面** |
| **W2** | **掌厅与加固** | 在 Redroid 内跑目标版本；记录 **崩溃/白屏/风控** 与 **ingest 无关** 的排障结论 | **一条固定业务路径** 在 SFU 出画下可重复完成 |
| **W3** | **多实例与端口（首版可 1 实例）** | 多路时 **每路 PlainTransport 端口/RTCP** 不与现有 compose、**`MEDIASOUP_RTC_*`** 冲突；脚本或文档写清 **重启后如何取新端口** | 同机 **≥2 路 ingest**（若需要）或明确 **首版仅 1 路** |
| **W4** | **镜像与发布** | 改 **`server.cjs` / ingest 侧** 后 **`docker compose build`** 习惯固化（见 `layer-c1-lessons-learned.md`） | 宿主机与容器内 **`server.cjs` 指纹一致** |

### 7.2 浏览器与入口（HTTPS + 信令 + consume）

| 顺序 | 工作项 | 要做什么 | 产出 / 验收 |
|------|--------|----------|-------------|
| **W5** | **产品页** | 在现有 **`mediasoup-client`** 能力上：**抽组件 / 换 UI**；**仅观看** 或 **固定 join 房间** 与业务一致 | 外网打开 **HTTPS 页面** 即 **RecvTransport + consume** 出画 |
| **W6** | **TLS 与域名** | **Caddy / Nginx** 终结 **`https://域名`**；**WSS 信令** 与页面同源或明确跨域；`docs/aliyun-ecs-pilot.md` §4.2 | 手机 4G 可开页、**信令 WSS 通** |
| **W7** | **鉴权** | Token / 短链 / 服务端校验；**joinRouter** 或等价 RPC 与 Token 绑定 | 无 Token **不可 consume** |

### 7.3 控制与触控（与媒体栈接口清晰）

| 顺序 | 工作项 | 要做什么 | 产出 / 验收 |
|------|--------|----------|-------------|
| **W8** | **会话绑定**（**可并行 W5～W7**） | **Token / 会话 ID → 唯一 Redroid + 唯一 ingest 端口上下文**；断线回收 | 刷新/重进 **行为可定义**（首版可「需重新取链」） |
| **W9** | **C2** | 浏览器坐标 → **HTTP/WebSocket API** → **`adb shell input`**；与 **W5 视频分辨率** 映射一致 | **§6.3** 中「可操作掌厅」项可打勾 |

### 7.4 观测与里程碑①收口

| 顺序 | 工作项 | 要做什么 | 产出 / 验收 |
|------|--------|----------|-------------|
| **W10** | **指标** | 首帧耗时、consume 失败率、ingest 断流次数、单会话 CPU/内存 | 对齐阶段 0 SLO 初值 |
| **W11** | **§1.1 第一步书面验收** | 对照 **§6.3** 全表 + 文首 **技术栈冻结** 自查 | 签字/记录日期后进入 **§1.1 第二步（1b 本地单机压测）** |

### 7.5 明确不做（避免栈漂移）

- **不**把 **WSS + 独立投屏解码** 做成与 **WebRTC consume** 并列的 **主出画路径**。  
- **不**在未 **`build`** 的情况下假设 **`server.cjs` / ingest 逻辑** 已进线上镜像。  
- **不**在 **W1～W2** 未稳时扩散 **多池、多机**（留给 **§1.1 第三步**）。
