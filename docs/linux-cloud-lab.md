# Linux 云机实验指南

在 **Linux 云主机（ECS / VPS）** 上跑本仓库实验，比在 Mac Docker Desktop 上更接近 **Redroid + mediasoup** 的真实部署形态。本文只列**可操作清单**；Redroid 与 ADB 隧道见根目录 **`README.md`**、`docker-compose.yml` 与 **`docs/redroid-notes.md`**。

**阿里云 ECS 分步清单（先跑通再扩展）**：见 **[aliyun-ecs-pilot.md](aliyun-ecs-pilot.md)**。  
**x86 本地 Linux（与云上同一套 Docker/compose，网络/KVM 差异）**：见 **[local-x86-linux.md](local-x86-linux.md)**。

## 1. 云机建议配置

- **系统**：Ubuntu 22.04 / 24.04 等常见 LTS（内核尽量新，Redroid 见 [redroid-doc](https://github.com/remote-android/redroid-doc)）。  
- **Docker**：Engine + Compose plugin；**启用 `network_mode: host`** 的能力（Linux 一般具备；**非** Mac Docker Desktop）。  
- **网络**：固定 **公网 IP** 或明确 **内网访问路径**（浏览器从哪条路径连到云机，后面 `MEDIASOUP_ANNOUNCED_IP` 就要填「对端能路由到的」那一侧）。

## 2. 安全组 / 防火墙（必做）

按你实际要开的端口调整；以下为 **WebRTC SFU 试点**（`experiments/webrtc-sfu-pilot`）与 **SSH** 的常见组合。

| 方向 | 协议 | 端口 | 用途 |
|------|------|------|------|
| 入站 | TCP | 22 | SSH（管理） |
| 入站 | TCP | 3000（或你设的 `PORT`） | H5 + WebSocket 信令（HTTP） |
| 入站 | UDP | `MEDIASOUP_RTC_MIN_PORT`–`MEDIASOUP_RTC_MAX_PORT`（默认 **40000–49999**） | mediasoup WebRTC 媒体面 |

- 若暂时只想收窄 UDP 范围，可在启动前设置 **`MEDIASOUP_RTC_MIN_PORT` / `MEDIASOUP_RTC_MAX_PORT`**（与云厂商安全组一致）。  
- **Redroid**：根目录 `docker-compose.yml` 默认把 **5555 绑在 `127.0.0.1`**，不直接对公网；ADB 通常走 **SSH 端口转发**（见 `docs/redroid-notes.md`）。若你自行改成 `0.0.0.0:5555`，须单独评估安全风险。

**ufw 示例**（端口与策略请按环境修改）：

```bash
sudo ufw allow 22/tcp
sudo ufw allow 3000/tcp
sudo ufw allow 40000:49999/udp
sudo ufw enable
```

## 3. 实验 A：仅跑 WebRTC SFU 试点（Layer B）

在云机上克隆本仓库后：

```bash
cd experiments/webrtc-sfu-pilot
npm install
npm run build:client
export MEDIASOUP_ANNOUNCED_IP=<浏览器访问云机时使用的 IP>
# 若浏览器用公网域名解析到弹性 IP，则填该弹性 IP（或等价可达地址）
export PORT=3000
node server.cjs
```

或用 **Docker（host 网络）**：

```bash
cd experiments/webrtc-sfu-pilot
export MEDIASOUP_ANNOUNCED_IP=<同上>
docker compose up --build
```

验证：本机或另一台机浏览器打开 `http://<云机IP>:3000/`，**两个 Tab**：一 Tab「发布摄像头」，另一 Tab「仅观看」。  
自动化检查（可在云机或 CI 上跑）：仓库根目录执行 `bash scripts/smoke-webrtc-sfu-pilot.sh`（默认随机端口；云机可 `PORT=3000` 固定）。

详见：`docs/webrtc-sfu-pilot.md`。

## 4. 实验 B：Redroid 云安卓（与 SFU 独立）

与 **WebRTC 试点**可并行装在同一台云机，但 **当前仓库未把「安卓画面」自动接入 SFU**（Layer C 仍待设计）。

```bash
# 仓库根目录

# ADB 经 SSH 转发等，见 docs/redroid-notes.md
```

## 5. `MEDIASOUP_ANNOUNCED_IP` 怎么填

- 浏览器地址栏是 `http://1.2.3.4:3000` → 一般填 **`1.2.3.4`**（与 ICE 对外宣告一致）。  
- 仅内网访问（同一 VPC 内另一台机打开页面）→ 填 **内网 IP**。  
- 填错或未填：常见现象是 **信令能连、页面能开，但无视频或 ICE 失败**。

## 6. 与「商用云手机」的差距（预期管理）

- **Layer B**（本仓库已提供）：浏览器摄像头经 mediasoup 转发，用于验证 **SFU + 信令 + 防火墙**。  
- **Layer C**（未提供）：Redroid / 真机 **屏幕编码 → WebRTC Producer**、以及 **触控回注**，需单独架构与开发，不能由 Layer B 自动替代。

## 7. 采购话术 vs 工程现实（ARM / x86 / KVM /「并发」）

网上常见「照抄购买 + 一键脚本 + 10086 绝不卡启动 + 20 万并发」类文案，与本仓库**可验证能力**混用容易误判。下面只列**技术事实**（不构成采购建议或 SLA）。

### 7.1 ARM 有没有物理机？

**有。** ARM 服务器（鲲鹏/飞腾等）、工控与板卡（如部分 RK 方案）、云厂商 ARM 实例，都属于 ARM 形态，**用途与成本差异极大**，不能互相替代结论。是否选 ARM 实例跑 Redroid，以 **内核 / 镜像 / 厂商说明** 与实测为准。

### 7.2 x86 自建机还能不能跑？

**可以。**在 **Linux** 上跑 Redroid 的常见前提是 **内核与特权容器** 等条件满足；许多资料会写 **KVM / 嵌套虚拟化** 与 **x86 云机** 的兼容性注意点，但「**x86 + 开 KVM = 10086 必不卡启动**」属于**过度承诺**——应用仍可能因**加固、风控、网络、ROM 差异**失败，与你在模拟器/容器里见过的日志同类问题**不会**因一句「上物理机」自动全部消失。

### 7.3「20 万并发」与「要多少台机器」

**必须先定义「并发」指什么：**

| 指标类型 | 瓶颈大致在哪里 | 备注 |
|----------|----------------|------|
| 20 万 **浏览器同时观看**（多路 WebRTC 下行） | **SFU 集群、出口带宽、CDN/边缘、TURN、信令** | 与「单台能跑多少 Redroid」是不同维度；不能用「实例数 ÷ 单机密度假算」定架构。 |
| 20 万 **云手机实例 / 会话** | **编排、镜像、存储 IO、GPU/编码、调度、数据库** | 与 mediasoup 单机配置是两套系统。 |
| 20 万 **HTTP 请求/秒** | 接入层、业务服务 | 与 WebRTC 媒体面不同。 |

因此像「**200000 ÷ 100 = 2000 台物理机**」这类**单一除法**，没有工程上的标定意义；真实容量需要 **压测模型 + 业务指标 + 失败降级策略**。

### 7.4 与本仓库的对应关系

- **阿里云 / 自建**：只要满足 **Linux + Docker（及 Redroid 文档要求）**，都可以作为 **实验 B（Redroid）** 或 **实验 A（SFU 试点）** 的运行环境。  
- **10086**：能否稳定启动属于 **App + 环境 + 合规** 问题，**不在**本仓库承诺范围内。  
- **Redroid + MediaSoup + H5**：本仓库当前提供 **Redroid compose（根目录）** + **mediasoup 试点 Layer B**；二者**尚未**自动接成「安卓画面进 SFU」（见 §6、 `docs/webrtc-sfu-pilot.md`）。

### 7.5 不推荐盲信的做法

- **`curl … | bash` 来源不明的「一键部署」**：有供应链风险，生产环境应使用**可审计**的镜像与脚本。  
- **把「轻量应用服务器 / 某规格」一棍子打死**：是否适合 Redroid，取决于**内核、嵌套虚拟化、厂商限制**与实测；以厂商文档与 PoC 为准。  
- **Mac 上嵌套 Linux 再跑 Redroid**：往往折腾且与生产路径不一致；**实验优先用 Linux 云机或真 Linux 主机**更省事（见本文开头）。

## 8. 白话：三种「并发」是啥？你要的「二十万云手机实例」又怎么理解？

下面用**不看 WebRTC 细节也能懂**的说法，对照你关心的 **「二十万个云手机实例并发」**。

### 8.1 三种口径各像什么？

| 口径 | 白话比喻 | 你在优化什么 |
|------|-----------|--------------|
| **二十万路「看别人画面」**（浏览器看 WebRTC） | 二十万人在家里**看电视转播**——大家看的是「已经采好、编码好、从服务器送出来的流」 | **转播网**：SFU、带宽、CDN、TURN、信令。和「机房里有二十万台真手机」不是一回事。 |
| **二十万个「云手机实例」** | 二十万台**同时开机、系统里跑着安卓环境**（每实例占内存/CPU/磁盘配额） | **机房里的手机池**：单机一台能「养」几个实例、镜像怎么发、盘够不够快、调度器怎么分配、挂了怎么换机。 |
| **二十万 HTTP 连接/秒** | 二十万次/秒有人**按门铃问一句**（请求很短） | **网站/API 网关**：和「每台手机里 App 在跑」不是同一套系统。 |

所以你选 **「二十万云手机实例」** = 你关心的是**第二行**（手机池），不是电视转播那条线。

### 8.2「二十万云手机实例」在合同/架构里还要钉死什么？

「实例」在工程里**仍不止一种数法**：

1. **同时常驻、可交互**（用户随时能点、进程都在跑）——最吃内存，**单机能跑的个数最少**。  
2. **挂着但冻结/休眠**（占磁盘或占元数据，不占满内存）——**同一台物理机可「名义上」挂更多**，但用户唤醒时要等冷启动。  
3. **峰值注册二十万、同时在线只有五万**——容量按**同时在线**算，不是按注册总数。

**你已明确口径：二十万 = 第 1 种「同时常驻、可交互」。** 这是**资源最紧**的一种；对外沟通时建议写进 SLA，避免被换成 2/3。

在同一口径下，仍要钉死 **「每台实例至少占多少资源」**（否则仍无法换算宿主机台数），例如：

- **每实例预留内存**（例如空系统基线 vs 预装目标 App 后的工作集）；  
- **CPU 份额**（vCPU 绑定或 cgroup 配额）；  
- **可写磁盘 / IOPS**（每实例上限）；  
- **是否允许 swap**（允许会「显得」能多塞实例，但交互延迟与抖动会上去）。

没有「每实例资源下限」，**200000 仍无法换算成机器台数**。

### 8.3 在「二十万同时常驻」下，为什么不能「除一下」就得到机器台数？

因为每台宿主机能跑的实例数**不是常数**，至少取决于：

- **单实例内存/CPU**：空桌面 vs 开着掌厅、是否预装大 App、是否允许 swap。  
- **存储**：每个实例的数据层、镜像层、日志；**磁盘 I/O** 往往比「算力够」更早成为瓶颈。  
- **调度与隔离**：崩溃重启、热迁移、资源超卖比例（商业上常见，但会影响抖动）。  
- **网络**：二十万台机子若还要**出网**（更新、广告、业务 API），出口和 NAT 又是另一套容量。

所以**最低成本要多少机器** = 在你**钉死 SLA**（**同时常驻 20 万** + **每实例内存/CPU/盘/网下限** + 镜像与目标 App）之后，用 **PoC 单机密度 × 冗余系数 × 调度损耗** 去**压测估区间**；不是一句除法能替你做采购决策。

### 8.4 数量级直觉（仅帮助理解规模，不是密度承诺）

在**「二十万同时常驻」**下，宿主机集群的 **RAM 总需求**会最先暴露「是不是天文数字」——只作**算术示意**，真实可跑密度以压测为准：

- 若你业务上要求 **每实例平均独占约 2 GiB 内存**（仅作假设），则 **200000 × 2 GiB ≈ 390 TiB 量级**的内存总预算（未计宿主机内核、容器开销、冗余）。  
- 实际 Redroid 单实例基线可能低于或高于 2 GiB，但一旦叠加 **目标 App（如掌厅）工作集**，往往会上浮；**共享只读镜像**能省盘、**省不了**每实例各自的用户态与缓存占用的大头。

因此：**「二十万同时常驻」在工程上一定是大型集群**；具体要多少台机，只能在你方 **PoC 定出「单机稳定 N 个实例」** 之后用 **200000 / N × 冗余** 估算，而不是反推一个常数 N 写进采购合同。

### 8.5 和你当前仓库的关系

本仓库帮你验证的是：**单/少量 Redroid**、**少量浏览器经 mediasoup 收发媒体（Layer B）** 这类 **PoC**。  
**二十万同时常驻实例**需要的是：**编排平台、镜像分发、监控计费、多地域、容量压测与冗余** 等产品级能力，已超出本仓库默认交付范围。  
**对外口径**可固定为：**二十万 = 同时常驻、可交互；台数 = 压测得到单机稳定实例数 N 后换算，并单独定义每实例资源下限。**

### 8.6 必须跑掌厅时：每实例内存怎么估（分档 + 实测）

Redroid **没有**「单实例法定最小内存」的通用表；掌厅（重 **WebView / 网络 / Native**）比空桌面吃内存得多。下面给的是 **规划用分档**（用于 PoC 起点与粗算数量级），**不是** SLA 承诺；最终以你在 **目标镜像 + 目标版本掌厅 + 典型操作路径** 下的实测为准。

| 档位 | 每实例建议 **预留**（宿主机视角） | 典型含义 |
|------|-----------------------------------|----------|
| **下限（高风险）** | **约 1.5–2 GiB** | 能完成「安装 → 冷启动 → 进首页」的概率较高，但 **OOM / 频繁重载 / 一切后台就杀** 风险大；不适合作为「二十万同时常驻」的默认规划值。 |
| **常用 PoC 起点** | **约 2.5–3 GiB** | 多数团队做「常驻可点」的第一轮压测会从这里起；仍要在你的镜像上验证。 |
| **更稳（推荐做预算上限侧）** | **约 3–4 GiB** | WebView 多页、图片缓存、活动页常驻时更不容易抖；单机实例数会变少，但可避免为省内存导致大面积不可用。 |

**为什么只能估区间：** 同一掌厅版本在不同 **Redroid/Android 大版本**、是否带 **GMS**、分辨率/DPI、是否预装其他服务、用户是否长时间停留在重页面上，**PSS / 峰值 RSS** 都会变。

**建议在阿里云上 1 台 PoC 机实测（1～2 天能做完）：**

1. 起 **1 个** Redroid，安装 **官方渠道掌厅**，走通你的 **典型用户路径**（冷启 → 登录/首页 → 点进 1～2 个常用二级页 → 静置 5～10 分钟）。  
2. 在宿主机看 **`docker stats`**（或 cgroup）里该容器的 **MEM USAGE / LIMIT** 峰值；同时在设备内 **`adb shell dumpsys meminfo com.greenpoint.android.mc10086.activity`**（包名以你实际为准）看 **TOTAL PSS** 与 **Native Heap**。  
3. 把 **峰值 × 1.2～1.5** 作为「单实例预留」的第一版（留内核抖动与短时尖峰）；再在同一台机上 **线性加实例**（2、4、8…）看从哪一档开始出现 **大量 lmkd / lowmemorykiller / 卡顿突增**。

**二十万同时常驻粗算（仅数学示意，用你实测 N 替换假设）：**

- 若 PoC 后认定 **每实例需预留约 3 GiB**，则仅内存总预算量级约为 **200000 × 3 GiB ≈ 586 TiB**（未计宿主内核、系统服务、冗余、非容器开销）。  
- 用你压出来的 **单机稳定实例数 N** 去除 **200000**，才进入「要多少台宿主机」的区间估算。
